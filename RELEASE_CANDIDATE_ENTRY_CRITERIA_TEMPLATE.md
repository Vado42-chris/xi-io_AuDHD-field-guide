# Release Candidate Entry Criteria Template

## Why this document exists

The repo now has:
- a pilot readiness checkpoint
- a pilot readiness progression ladder
- a broader readiness planning checkpoint
- batch workbooks, closeouts, and comparisons
- a cumulative trusted pilot report
- a master progress and release posture ledger

That is enough to define pilot posture and broader-readiness posture clearly.

The next practical need is an explicit **release-candidate entry criteria document**.

This document is not the release candidate itself.
It is the gate that answers:
- what must already be true before RC-track work can begin
- what evidence must exist first
- what blockers automatically disqualify RC posture
- what remains pilot-only work versus true release-track work

This checkpoint exists to prevent release language from outrunning product proof.

# 5. Required evidence before RC-entry

Check only what is genuinely supported by evidence.

- [ ] trusted pilot evidence exists beyond planning and templates

- [ ] broader readiness planning has been evaluated explicitly
- [ ] the core support loop has shown stable enough behavior across evidence
- [ ] Help Now has shown durable clarity and usefulness
- [ ] Learn Me has shown durable enough repeated-use value
- [ ] personalization has shown meaningful support value over time
- [ ] trust language has remained appropriately cautious and understandable
- [ ] regression work has been executed and reviewed
- [ ] major blockers are no longer dominating the signal
- [ ] integration confidence is strong enough that RC-track work is not purely theoretical

# 12. Honest reminder

RC-entry is not a reward for having a good-looking pilot system.
It is a decision that the product has accumulated enough real evidence, enough real stability, and enough real blocker reduction that release-track work becomes responsible to begin.

If that evidence is not there, this checkpoint should say so plainly.
